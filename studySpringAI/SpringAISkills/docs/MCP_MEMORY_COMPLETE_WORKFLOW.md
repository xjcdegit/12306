# SpringAISkills 完整工作流程设计文档

## 概述

本文档整合了 **MCP远程调用服务** 和 **上下文记忆系统** 两大核心功能，构建一个完整的智能Agent工作流程。

## 二、核心概念定义

### 2.1 MCP协议层

| 概念 | 层次 | 职责 |
|------|------|------|
| **MCP** | 协议层 | 标准化的工具调用协议，负责传输 |
| **原子 Tool** | 协议层 | 最小操作单元，如查询数据库、调用 API |
| **Skill** | 业务层 | 工作流，封装多个原子 Tool 完成业务目标 |
| **Agent** | 调度层 | 管理 LLM 与 Skill 之间的交互循环 |

### 2.2 记忆系统层

| 记忆类型 | 存储位置 | 容量 | 用途 |
|----------|----------|------|------|
| **短期记忆（L1）** | JVM内存 | 最近10-20轮对话 | 当前会话连续对话 |
| **长期记忆（完整）** | MySQL数据库 | 无限制（智能压缩） | 完整历史存档、审计 |
| **压缩摘要** | 数据库 | 按需生成 | 保留关键历史信息 |

## 三、完整工作流程

### 阶段 0：会话初始化与记忆加载

```
用户发起请求（携带 conversationId + userId）
    ↓
Agent 检查会话状态
    ↓
┌─────────────────────────────────────────┐
│           记忆系统查询                    │
├─────────────────────────────────────────┤
│ 1. 查询短期记忆（InMemoryChatMemory）    │
│    └─ 返回：最近N轮对话                  │
│                                          │
│ 2. 查询长期记忆（RelationalDatabaseMemory）│
│    ├─ 返回：压缩摘要（历史关键信息）      │
│    └─ 返回：最近未压缩消息               │
│                                          │
│ 3. 合并构建完整上下文                    │
│    └─ 压缩摘要 + 最近消息                │
└─────────────────────────────────────────┘
    ↓
构建 Message 数组（历史对话 + 当前输入）
```

### 阶段 1：Agent 接收请求

```
接收输入：用户通过前端界面发送自然语言指令
    ↓
解析参数：
    ├─ conversationId: 会话唯一标识
    ├─ userId: 用户唯一标识
    └─ message: 用户自然语言输入
    ↓
加载会话历史（从记忆系统）
```

### 阶段 2：获取 MCP 工具集

```
Agent 需要知道当前系统具备哪些能力
    ↓
并行广播：Agent 作为 MCP Client，向所有 MCP Server 发送 tools/list
    ↓
┌─────────────────────────────────────────┐
│        MCP Server 连接池                 │
├─────────────────────────────────────────┤
│  SSE 连接：                              │
│  ├─ 高德地图 MCP: https://mcp.amap.com/sse│
│  └─ 本地 MCP: http://localhost:8000/sse  │
│                                          │
│  STDIO 连接：                            │
│  ├─ Microsoft Learn MCP                  │
│  └─ Fetch MCP                            │
└─────────────────────────────────────────┘
    ↓
聚合工具池：各个 MCP Server 返回工具元数据
    ↓
内存缓存工具列表（避免重复请求）
```

### 阶段 3：构建大模型载荷

```
Agent 组装发送给 LLM 的最终请求包
    ↓
┌─────────────────────────────────────────┐
│           载荷构建                       │
├─────────────────────────────────────────┤
│ 1. 注入 System Prompt                   │
│    └─ 设定 Agent 人设与硬性约束          │
│                                          │
│ 2. 拼接 Message 数组                     │
│    ├─ 历史摘要（压缩后的关键信息）        │
│    ├─ 最近对话（短期记忆）               │
│    └─ 当前用户输入                       │
│                                          │
│ 3. 挂载 Tools Schema                    │
│    └─ MCP 工具转换为 tools 数组          │
└─────────────────────────────────────────┘
    ↓
发送请求到 LLM
```

### 阶段 4：LLM 决策推理

```
大模型进行意图识别和规划
    ↓
┌─────────────────────────────────────────┐
│           LLM 推理过程                   │
├─────────────────────────────────────────┤
│ 1. 意图匹配                             │
│    └─ 分析用户需求                       │
│                                          │
│ 2. 工具匹配                             │
│    └─ 扫描 tools 数组，匹配所需工具      │
│                                          │
│ 3. 决策输出                             │
│    ├─ 直接回答（无需工具）               │
│    └─ tool_calls（需要调用工具）         │
└─────────────────────────────────────────┘
```

### 阶段 5：MCP 路由与物理执行

```
真正的业务代码执行阶段
    ↓
┌─────────────────────────────────────────┐
│        MCP 工具调用流程                  │
├─────────────────────────────────────────┤
│ 1. 拦截与路由                           │
│    └─ 解析工具名称，查找归属 MCP Server  │
│                                          │
│ 2. 触发 MCP 调用                        │
│    └─ 发送 tools/call JSON-RPC 请求     │
│                                          │
│ 3. 物理执行                             │
│    └─ MCP Server 执行底层代码           │
│                                          │
│ 4. 结果返回                             │
│    └─ Server 将结果打包成 JSON 传回     │
└─────────────────────────────────────────┘
    ↓
工具执行结果
```

### 阶段 6：闭环回传与记忆存储

```
Agent 将执行结果回传给 LLM
    ↓
┌─────────────────────────────────────────┐
│        结果处理与记忆存储                │
├─────────────────────────────────────────┤
│ 1. 追加工具结果                         │
│    └─ 封装成 role: tool 消息            │
│                                          │
│ 2. 存储到记忆系统                       │
│    ├─ 短期记忆：添加到内存队列           │
│    └─ 长期记忆：持久化到数据库           │
│                                          │
│ 3. 检查压缩触发条件                     │
│    └─ token数 > 阈值 → 触发压缩         │
│                                          │
│ 4. 二次请求 LLM                         │
│    └─ 将 messages 数组发送给 LLM        │
└─────────────────────────────────────────┘
    ↓
LLM 判断任务是否完成
    ├─ 未完成 → 循环回阶段5（再次 tool_calls）
    └─ 已完成 → 生成最终响应
    ↓
返回用户
```

## 五、记忆系统与MCP协同工作

### 5.1 记忆系统架构

```
┌─────────────────────────────────────────────────────────────┐
│          记忆系统架构                                         │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  短期记忆（L1）                                               │
│  ├─ 实现：InMemoryChatMemory                                 │
│  ├─ 存储：JVM内存                                            │
│  ├─ 容量：最近10-20轮对话                                    │
│  └─ 用途：当前会话连续对话                                    │
│                                                               │
│  长期记忆（完整）                                             │
│  ├─ 实现：RelationalDatabaseMemory                           │
│  ├─ 存储：MySQL数据库                                        │
│  ├─ 容量：无限制（智能压缩）                                  │
│  ├─ 压缩：学习Claude Code，自动压缩旧消息                     │
│  └─ 用途：完整历史存档、审计                                  │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 智能压缩机制

```
当token数量超过阈值（默认8000）
    ↓
保留最近10条消息不压缩
    ↓
旧消息通过LLM生成摘要
    ↓
删除旧消息，保存摘要
    ↓
查询时返回：压缩摘要 + 最近消息
```

### 5.3 压缩流程图

```
新消息进入
    │
    ▼
计算总token数
    │
    ▼
┌─────────────────┐
│ token数 > 阈值? │
└─────────────────┘
    │
    ├── 否 ──→ 正常存储
    │
    └── 是 ──→ 触发压缩
                  │
                  ▼
           获取未压缩消息
                  │
                  ▼
           保留最近10条
                  │
                  ▼
           剩余消息生成摘要
                  │
                  ▼
           保存摘要记录
                  │
                  ▼
           删除旧消息
                  │
                  ▼
           压缩完成
```

## 九、数据流转示意

### 9.1 一次完整对话的数据流

```
1. 用户输入
   {
     "conversationId": "conv-001",
     "userId": "user-001", 
     "message": "北京今天天气怎么样？"
   }
        │
        ▼
2. 记忆加载
   短期记忆: [最近5轮对话]
   长期记忆: [压缩摘要] + [最近10条消息]
        │
        ▼
3. 构建LLM请求
   {
     "messages": [
       {"role": "system", "content": "你是智能助手..."},
       {"role": "assistant", "content": "【历史摘要】用户之前询问过..."},
       {"role": "user", "content": "北京今天天气怎么样？"}
     ],
     "tools": [
       {"name": "amap_weather", "description": "查询天气", ...}
     ]
   }
        │
        ▼
4. LLM返回tool_calls
   {
     "tool_calls": [
       {"name": "amap_weather", "arguments": {"city": "北京"}}
     ]
   }
        │
        ▼
5. MCP调用
   POST https://mcp.amap.com/sse
   → 返回: {"temperature": "25°C", "weather": "晴"}
        │
        ▼
6. 结果回传LLM
   {
     "messages": [
       ...历史消息...,
       {"role": "tool", "content": "{\"temperature\": \"25°C\"}"}
     ]
   }
        │
        ▼
7. LLM生成最终响应
   "北京今天天气晴朗，气温25°C，适合外出活动。"
        │
        ▼
8. 存储到记忆系统
   短期记忆: 添加本轮对话
   长期记忆: 持久化到数据库
        │
        ▼
9. 检查压缩触发
   token数 < 8000 → 不压缩
   token数 ≥ 8000 → 触发压缩
        │
        ▼
10. 返回用户
    {"result": "北京今天天气晴朗，气温25°C，适合外出活动。"}
```

## 十一、职责边界

| 模块 | 职责 | 不负责 |
|------|------|--------|
| **MCP** | 协议传输、工具发现、工具调用 | 业务逻辑实现 |
| **记忆系统** | 上下文存储、历史检索、智能压缩 | 工具调用决策 |
| **Agent** | 调度协调、LLM交互、流程控制 | 具体业务实现 |
| **Tool** | 原子操作执行 | 流程编排 |

## 十三、总结

本工作流程整合了 **MCP远程调用** 和 **上下文记忆** 两大核心能力：

### 核心特性

| 特性 | 描述 |
|------|------|
| ✅ MCP远程调用 | 标准化工具协议，支持SSE/STDIO连接 |
| ✅ 短期记忆 | JVM内存存储，快速访问最近对话 |
| ✅ 长期记忆 | 数据库持久化，完整历史存档 |
| ✅ 智能压缩 | 自动压缩旧消息，保留关键信息 |
| ✅ 会话管理 | conversationId + userId 双维度管理 |

### 工作流程要点

1. **会话初始化**：加载历史记忆构建上下文
2. **工具发现**：并行请求MCP Server获取工具列表
3. **智能决策**：LLM根据上下文和工具能力做出决策
4. **工具执行**：通过MCP协议调用远程服务
5. **记忆存储**：持久化对话记录，触发智能压缩
6. **响应生成**：结合工具结果生成最终回复
